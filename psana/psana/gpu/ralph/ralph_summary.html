01psana2 · GPU Jungfrau calibration · LCLS-II

What earned its place —
and what the benchmarks killed

A 29-iteration autonomous benchmark campaign (the “Ralph loop”, July 10–13, 2026) took the GPU calibration MVP from first anchor measurements to a measured endpoint: storage-bound, code-side levers exhausted. Along the way it retroactively audited every feature deferred out of the original branch — confirming two, killing eight, and pricing the revival criteria for the rest.

02what proved useful · the core

462 lines beat a path that can never catch up

The MVP is deliberately tiny: a fused CUDA kernel plus a two-function API, called from psana’s standard event loop. No custom iterator, no DataSource integration, no wire format.

Code that carries the resultLines

gpu_calib.py — constants prep + launch186
__init__.py — GPU rank init83
test_jungfrau_calib.py — bit-exact gate138

The mechanism, not just the ratio: CPU calibration inflates from 31.6 to 433.7 ms/event (13.7×) under 32-way memory-bandwidth contention. That caps the CPU path at ~74 Hz per node — at any rank count. The GPU path doesn’t just win; it removes the ceiling.

03what proved useful · landed throughput wins

Two nearly-free changes: stop copying 33.5 MB per event

Profiling (not intuition) found that the GPU path spent most of its CPU time on two host-side memcpys of the full detector frame. Both fixes are small, both were gated bit-exact, and both were measured with interleaved A/B brackets in the same allocation.

at 1 BD and 32 BD. det.raw.raw(evt, copy=False) — the flag already existed; the GPU path consumes the array immediately, so the defensive copy was pure waste. Promoted to the GPU-path default.
iter 8 · per-segment H2D
+38% / +32%
1 BD / 32 BD. Skip the host stack memcpy: 32 per-segment 1 MB transfers straight into a device buffer (raw_gpu_buf[idx].set(segs[sid].raw)) instead of assembling the frame on host first.
combined · single node, 32 BD
+70%
210 → 337 Hz across the two landed wins. Absolute Hz varies with FFB window (live production filesystem); the interleaved relative gains are the controlled numbers.

Pattern worth keeping: every landed win here removed work from the per-event path. Every attempt to overlap or hide work instead (async H2D, D2H every event) lost to memory-bandwidth contention at scale. On a contended node, bytes moved is the budget.

04what proved useful · multi-node

~120 lines of EB placement broke the 300 Hz plateau

Multi-node runs plateaued near 300 Hz regardless of node count. The cause wasn’t storage — it was a single global EventBuilder serializing as it served more BD ranks (eb_wait climbing 3.5 → 70 ms as BDs went 32 → 96). Two opt-in, default-off env flags fixed it:

  • PS_EB_NODE_LOCAL=1 — one EB per physical node, serving only that node’s BD ranks (psexp/node.py, ~48 lines changed).
  • PS_EB_PER_NODE=k — sub-split each node into k EventBuilders. The knee is k=2 at 2 nodes, k=3 at 3 and 4 nodes — confirmed in both warm-up orders; past the knee, extra EBs starve BD-reader slots (node.py + datasource.py, ~36 lines).

Default path untouched: both flags are opt-in and gated entirely behind env vars — unset, the standard psana MPI event loop is byte-for-byte unchanged.

05method · the ralph loop

The campaign engine: a memoryless agent, one proof per turn

A bash driver runs up to 25 iterations. Each iteration is a fresh agent with no memory of the previous one — its entire state is what the journal and the repo say. That forcing function is the design: nothing counts unless it was written down and committed.

  • 29 iterations over 4 days, typically 4–30 turns and ~$1–6 each; the human reviewed the journal between blocks and restarted with sharpened orders.
  • Self-terminating by evidence: the loop may only write LOOP DONE after a pre-registered kill criterion is closed by measurement — it ended when the last untested code lever was measured and lost.
  • Negative results are kept, not discarded: dead-end levers stay in the tree as opt-in flags with VERDICT: DEAD END docstrings, so nobody re-derives them.
06method · why the verdicts hold

The discipline that makes “fluff” a safe word

Calling a deferred feature dead is a strong claim on a filesystem with minute-to-minute variance. The loop earned it with controls:

Bit-exact gate, every change

Every code change re-ran the correctness test: 20 events vs det.raw.calib(), max_diff 0.0 required. No throughput number was accepted from a path that hadn’t passed.

ABBA warming brackets

FFB warms up across a session, so A/B comparisons ran A-B-B-A in one allocation. A lever only won (or lost) if it beat the opposing phase in both positions — e.g. async-H2D lost even to the coldest baseline phase.

Reversed-order confirms

Any result that could be order-confounded was re-run with the phase order reversed (the k=3 vs k=4 knee at 4 nodes: k=3 won cold-first and warm-last).

Attribution counters

The bench splits per-event wall into bd_read / eb_wait / H2D / kernel / plumbing buckets — so claims name the component, not just the total. A same-denominator counter (iter 28) caught a 45.9 ms “residual” that turned out to be an artifact, not a lever.

Pre-registered kill criteria

Before running an experiment the journal states what result would kill the hypothesis. Iter 27/28 wrote the loop’s own exit condition — “only if this lever fails to move the aggregate is the loop done” — one iteration before it fired.

Raw-ceiling controls

A psana-free raw os.pread harness measured what the filesystem itself can do (~7.9 GB/s/node, scaling with nodes) — the yardstick that separates “psana inefficiency” from “physics of the storage”.

07the endpoint

Where the 103 ms per event actually goes

Final attribution at 32 BD ranks (iter 28, same-denominator counters). This chart is the audit criterion for everything on the next slides — a deferred feature can only matter if the component it accelerates is visible here.

Amdahl, concretely: make everything except the read infinitely fast and you gain ~8%. The closing experiment (iter 29) proved the corollary — overlapping the ~3.5 ms of GPU work behind the read regressed −21%, because its staging copy cost more host bandwidth than the time it hid.

08deferred audit · killed by measurement

Fluff, confirmed — each with its cause of death

These were deferred on judgment; the campaign upgraded that to evidence. Each card names the measurement that killed it, and the concrete check that would revive it.

EventPool / StreamPool fluff · confirmed

Killed by: occupancy math + kernel timing. One Jungfrau event launches 65,536 blocks vs the A100’s ~864-block saturation (7,585%); the kernel runs 0.6–1.0 ms. There is nothing for N-in-flight pipelining to fill.

Revives when: a detector whose single event under-fills the GPU needs this path. Check: single-event SM utilization < 50% (blocks-launched vs SM count) — i.e. small detectors at very high rate, not larger ones.
GpuEvents custom iterator fluff · confirmed

Killed by: iter 28’s same-denominator counters. The standard event loop’s overhead the iterator would replace — generator advance + dgram + SMD parse — is ~3 ms of a ~103 ms event (~3%).

Revives when: plumbing ≥ ~20% of per-event wall. That requires storage ~10× faster first — check the gen_wait / dgram / smdparse counters after any storage upgrade before considering it.
GPUBAT1 wire format + Dgram.raw_descriptors() + SmdInfoExtractIter fluff · confirmed

Killed by: the same counters — metadata serialization and offset extraction never appeared in any profile (SMD parse ≈ 1.9 ms). The direct-read path they exist to feed lost its justification (next slide).

Revives when: the direct BD-read family revives (slide 09 criteria) — these three are its plumbing and only earn their place together with it.
gpudgramlite (on-device dgram parse) fluff · confirmed

Killed by: smd0 measured idle and CPU-side parse ≈ 1.9 ms/event. GPU-side routing solves a bottleneck that does not exist at any measured scale.

Revives when: CPU-side event routing is measured ≥ ~20% of wall — effectively never on this workload; would need ~50× storage speedup.
gpu_bd_read standalone pipeline superseded

Killed by: the standard loop reaching 89–97% of the raw storage ceiling — a bespoke 516-line parallel pipeline has ≤ 11% of headroom to chase. Its correctness tools were replaced by test_jungfrau_calib.py (bit-exact PASS).

Revives when: same criteria as the direct-read family — and even then, prefer extending the standard loop over a parallel implementation.
09deferred audit · the near-miss

The direct-read family almost earned its place — honesty requires saying so

Mid-campaign, the evidence genuinely favored reviving KvikIO/GDS and the direct BD-read path: at 2 nodes psana achieved only 65% of raw storage (9.9 vs 15.2 GB/s), and a ~300 Hz “psana plateau” bound before storage did. Had the loop stopped at iteration 16, these features would look justified.

Iterations 18–27 found the real cause: smd0→EB coordination topology, fixed by PS_EB_NODE_LOCAL + PS_EB_PER_NODE. That result retroactively kills the family — there is no psana-vs-raw gap left for a direct-read path to close.

KvikIO / GDS reader + direct BD-read path conditional

Today: H→D is 2.53 ms async of a ~103 ms event — GDS’s best case saves ~3%. GDS skips the CPU bounce; it does not make the storage faster, and the storage is the wall. S3DF Lustre also has no GDS (all prior benchmarks silently ran the CPU-fallback path).

Revives when — check all three: (1) the psana-vs-raw gap reopens (< ~80% of a raw fio/pread control at the same layout) after EB-placement tuning; (2) per-event read drops to ≲ 15 ms (local NVMe staging or a faster FFB tier), making the 33.5 MB host bounce a ≥ 20% cost; (3) the filesystem actually supports GDS — kvikio compat_mode == False, not the silent fallback.
10deferred audit · alive

Two triggers fired exactly as written — four wait on use cases

AsyncD2HJoiner trigger met · real lever

Fired: synchronous .get() under 32-rank PCIe contention costs 72 ms/event and drops aggregate −28% (236 → 170 Hz), ~73% additive. Never implemented — a 370-line design note only.

Earns its place: the moment any production workflow needs calibrated frames back on host at scale — worth ~+39%. Design constraint from iter 29: stage device-side (D2D) only; any extra host-side copy per event will lose to memory-bandwidth contention, exactly as the H2D-overlap experiment did.
IPC calib-constant sharing (gpu_mpi.py) trigger met · memory, not Hz

Fired (iter 5): each BD rank replicates ~384 MB of constants; 48 ranks OOM a 40 GB A100, 32 fit. But throughput is flat 16 → 32 BD — sharing buys packing density, not speed. This is real, tested code (527 lines + its own test) recoverable at d1972b0ac.

Earns its place: any topology needing > 32 BD ranks/GPU, or smaller-memory GPUs. Check: constants-per-rank × ranks vs GPU memory, or an observed OutOfMemoryError in the sweep.
GPUKernelRegistry untested · event-driven

Built for a second detector that doesn’t exist yet. No benchmark can fire this — a use case does.

Earns its place: a second detector type (ePix10k, ePixHR, …) needs GPU calibration in production. Until then, one kernel needs no registry.
DetectorRouter untested · event-driven

GPU+CPU segment combining for split readouts. Never exercised — every run was all-GPU.

Earns its place: a real run mixes GPU-path and CPU-path panels of the same detector (e.g. only some streams on the GPU rank). Correctness then requires its segment scatter.
GpuEventContext / GPUResult untested · adoption-driven

API sugar over a two-function interface. Zero performance content.

Earns its place: users demonstrably struggle to mix GPU and CPU detectors with the plain API. Evidence is support requests, not benchmarks.
DataSource(gpu_det=…) integration untested · adoption-driven

The full psexp wiring. Historical warning: its first incarnation broke mpi_ds and the eventbuilder extension at import time for everyone.

Earns its place: GPU calib is production-adopted and the explicit API is a measured adoption barrier. Requirement carried forward: type-stable — never silently return CuPy where NumPy is expected.
11closing

The slim-to-MVP bet is now vindicated by measurement

Of ~11,200 deferred lines, only two items had triggers that could fire — both fired exactly as the deferral doc predicted, and neither is a throughput lever for this workload. Roughly 90% of the speculative architecture targeted components that sum to under 8% of the per-event wall. Nothing is lost: everything is recoverable at commit d1972b0ac.
