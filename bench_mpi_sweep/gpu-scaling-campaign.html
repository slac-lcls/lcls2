01 · The system in one picture

Think of it as a restaurant

psana’s parallel mode is a pipeline of three roles feeding the GPU. The analogy holds up well in every result below.

smd0 (rank 0)
The host with the reservation book

One person reads the index of every event and hands out work orders. Everything flows through them.

EventBuilder
The expediter

Matches up the pieces of each event and deals batches out to the waiters.

BD ranks
The waiters

Fetch the actual data from storage (the pantry) and carry it to the kitchen. You can hire as many as you want.

GPU
The kitchen

Enormous. Cooks a full event in 4 ms. It spends most of its time waiting for food to arrive.

The whole campaign in one sentence: the kitchen was never the problem. First the pantry was too far away (storage), and once we fixed that, the single host reading the reservation book became the limit (smd0). Hiring more waiters, expediters, or kitchens does not help a restaurant with one host.

02 · The code under test

The API, how the GPU is used, and what the numbers mean

from psana import DataSource
from psana.gpu import init_gpu_rank
init_gpu_rank()                     # pin this MPI rank to a GPU (must run before cupy import)
import cupy as cp
from psana.gpu import prep_calib_constants, fused_calib_gpu

ds  = DataSource(exp='mfx101210926', run=387, dir=FFB_PATH)
run = next(ds.runs())
det = run.Detector('jungfrau')
peds_gpu, gmask_gpu = prep_calib_constants(det)   # once per run: constants go to GPU and stay

for evt in run.events():                        # the completely standard psana loop
    raw = det.raw.raw(evt)                        # CPU memory: 33.5 MB of uint16
    if raw is None: continue
    calib_gpu = fused_calib_gpu(cp.asarray(raw), peds_gpu, gmask_gpu)
    # calib_gpu: float32, on the GPU. Call .get() only if you need it on the CPU.

What the GPU actually does

One fused kernel, one thread per pixel. Each thread reads its uint16, takes the top 2 bits as the gain mode (00, 01, 11; 10 marks a bad pixel), subtracts the pedestal for that mode, multiplies by the gain mask for that mode, and writes a float32. The pedestal and gain arrays are uploaded once per run and stay resident, so the only thing that crosses to the GPU per event is the 33.5 MB raw frame.

Occupancy: why one event fills the GPU 76 times over

One thread per pixel in blocks of 256 means one event launches 65,536 blocks. An A100 has 108 SMs (the physical compute units) and each SM can hold 8 of these blocks at a time, so only 864 blocks run at once. The other 64,672 wait in line: one event is 76 back-to-back waves of full-GPU work.

That is what “7,585% occupancy” means, and it is the whole argument against the old branch’s EventPool: pooling events helps when single pieces of work leave silicon idle. Here a single event already packs the kitchen 76 times over, so scheduling a second event alongside it cooks nothing faster.

Where “270 events/s per GPU” comes from, and why it looks slow

It is measured, and 91% of it is the copy, not the compute. Per event: copy to GPU 3.37 ms + kernel 0.32 ms = 3.69 ms, and 1000 / 3.69 = 270 events/s.

The copy is the suspicious part: 33.5 MB in 3.37 ms is 10 GB/s, only ~40% of what the PCIe gen4 link can carry (~25 GB/s). The cause is not PCIe latency (that is microseconds); it is that we hand CuPy a regular pageable numpy array. CUDA cannot DMA from pageable memory, so it first copies the data into an internal pinned staging buffer, then transfers. That extra memcpy roughly halves the bandwidth, and because everything is synchronous, copy and compute never overlap. So 270 Hz is the ceiling of the naive copy path, not of the GPU:

Transfer pathEffective rateEvents/s per GPU
Today: pageable memory, synchronous~10 GB/s270 (measured)
Pinned host buffer, synchronous~25 GB/s~600
Pinned + copy/compute overlap~25 GB/s, kernel hidden~745 (PCIe gen4 wire limit)
Kernel only, data already on GPU735 GB/s of HBM traffic~3,100

Would GPUDirect Storage (GDS) raise the ceiling?

No. GDS does not make PCIe wider; the bytes still cross the same gen4 link into the GPU, so the per-GPU ingest ceiling stays ~745 Hz at this event size. What GDS changes is the route: today data crosses PCIe twice (storage adapter into host RAM, then host RAM into GPU) and burns host RAM bandwidth and CPU on staging. GDS makes it one DMA, storage adapter straight to GPU memory. That helps when the host is the congested resource, which our 32-workers-per-node runs hinted at. Raising the wire ceiling itself takes a different bus: PCIe gen5 doubles it to ~1,500 Hz, NVLink-attached hosts go further.

Perspective for prioritizing: every number in this table is per GPU, while the whole pipeline currently delivers ~300 Hz total because of smd0. The copy path is not the binding constraint yet.

  8. 7
    smd0 batch size · smd0_d2h.sbatch

    Can a config setting speed up smd0?

    for N in 1000 4000 16000: # events per smd0 batch; 1000 is the default PS_SMD_N_EVENTS=$N mpirun --bind-to none -x PS_SMD_N_EVENTS -n 34 python ... --dir <FFB>
    batch 1000 / 4000 / 16000: 254 / 254 / 242 Hz. Flat.
    ceiling #3: smd0→No. That was the last free knob. Going past ~300 Hz now requires changing psana2 itself (parallel index reading), and that work starts with profiling, not building.
  9. 8
    copying results back to CPU · smd0_d2h.sbatch

    Is bringing results back from the GPU free?

    mpirun --bind-to none -n 34 python ... bench_calib.py -n 200 --dir <FFB> # control mpirun --bind-to none -n 34 python ... bench_calib.py -n 200 --dir <FFB> --d2h # + .get() per event
    without readback: 236 Hz · with: 170 Hz (-28%) · per-copy: 12.5 ms alone, 72 ms with 32 workers sharing the bus
    trigger met→No. Naive readback costs 28% of throughput, and crowding makes each copy 6x worse. This justifies the one deferred feature the data called back: batched async readback (AsyncD2HJoiner). Keeping results on the GPU stays free.

06 · The lesson

Why building ahead of the data is expensive

The old branch wrote thousands of lines across pipelines, backends, caches, pools, and custom I/O before running a single controlled measurement. This campaign is the case study in why that pattern is dangerous.

1. Guesses get frozen into architecture

Every speculative feature is a bet about where the bottleneck will be. When the bet is wrong, the code does not just sit there harmless: it becomes the frame everyone reasons inside. The old branch bet on GPU-side scheduling and custom I/O. The real bottlenecks turned out to be a directory path and one Python process reading an index. No amount of GPU plumbing could have found or fixed either.

EventPool, built to juggle events on the GPU, measured slower than doing nothing: 60 Hz vs 100 Hz.

2. Unmeasured code produces numbers that mislead

Complex code invites shortcut benchmarks, and shortcut benchmarks lie. The old branch recorded a kernel time of 0.002 ms, which is physically impossible for 60 MB of data. It was timing the kernel launch, not the kernel. Its headline rates were recorded without noting rank counts or storage. Wrong numbers are worse than no numbers, because they steer the next person’s decisions and look exactly like results.

Reported: kernel 0.002 ms. Measured properly: 0.32 ms. A 160x fiction, in the direction that flatters the design.

3. The cost is asymmetric: fast to write, slow to unwind

Writing speculative code takes days. Catching up to it takes much longer, because every piece must be read, understood, tested, and proven unnecessary before it can be removed, and it tangles into core files along the way. Unwinding this branch meant reverting 21 files and 4,374 lines threaded through psana’s core. The replacement that actually works is about 300 lines and two functions, and two days of measurement answered every question the thousands of lines had assumed answers to.

4. Complexity hides the cheap win

The single biggest improvement of the entire effort, 14x, cost zero lines of code. A team staring at a sophisticated GPU pipeline does not go looking for a filesystem path, because the sunk complexity points attention at itself. The simplest version of the system is also the one where the real problem is visible.

The working rule: build the minimum that is provably correct, write down every idea you deliberately did not build next to the measurement that would justify it, then run the measurements. Here the data called back exactly one deferred feature (async readback), rejected the rest, and pointed at one the old branch never thought of (parallel smd0). That is the scorecard for guessing versus measuring.
07 · Next steps

Ranked by impact per effort

1 · today · zero code

Run production GPU jobs on FFB, ~8 workers per GPU

That configuration delivers 195 Hz per GPU with almost no waste (~34 Hz per worker). Add nodes for aggregate throughput: 306 Hz measured on four. This is deployment guidance, not a project.

2 · small build · +39% when results return to CPU

Keep results on the GPU where possible; build batched readback where not

Downstream GPU analysis pays nothing. Workflows that need frames back on the CPU currently lose 28% to naive copying (236 to 170 Hz). Batched async readback through pinned memory recovers it, and the design note already exists in DEFERRED.md.

3 · psana2 core work · unlocks >300 Hz

Profile smd0, then parallelize the index reading

First measure where rank 0 actually spends its time: parsing, reading, or sending. That decides the design. This is the only remaining lever that moves the ceiling, and until it moves, no downstream optimization can help.

4 · after step 3

Pinned-memory upload staging with copy/compute overlap

Raises the per-GPU ingest ceiling from 270 toward ~745 Hz (the PCIe gen4 wire limit; see section 02). Cheap, contained in the GPU module, already sketched in DEFERRED.md. Only worth building once smd0 can feed faster than 270 Hz per GPU.
